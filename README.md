# thesis-draft-writer

一个可复用的 Codex skill 仓库，用来支撑 AI 主导的理工科论文成品写作。它的目标不是把论文流程脚本化，而是给主 AI 一套统一的工作约束、审查机制和少量刚性工具，让论文围绕真实文献、真实代码、真实实验和可辩护结论持续推进。

## 核心思想

1. AI 是作者与总控，不是脚本操作员。
2. 开题报告先被 challenge，再决定论文路线。
3. 影响方法、实验和结论的核心文献必须通读。
4. 对可实验课题，真实实验与代码实现是硬要求，不是附属品。
5. 审查主要靠 reviewer subagents，脚本只做结构、占位和格式类硬检查。
6. `docx` 和 Word 细修属于交付层，不应压过实验与结论本体。

## 典型工作流

1. 读取开题报告、任务书或需求材料。
2. 执行 `Proposal Challenge Gate`，找出概念混淆、过度承诺、平台限制和可替代路线。
3. 明确当前机器是否真的能完成实验。
4. 筛选并通读核心文献。
5. 由主 AI 自主写代码、跑实验、补证据。
6. 用真实结果组织方法、实验、分析、结论与局限。
7. 让 fresh-context reviewer subagents 做旁路审查。
8. 最后再做 `docx` 组装与 Word 格式修整。

## 使用方式

在支持 skill 的 Codex 环境中调用 `thesis-draft-writer`。

推荐起手提示词：

```text
使用 `thesis-draft-writer` 基于当前目录中的开题报告启动论文工作。

先完成 Proposal Challenge Gate、平台可行性判断、核心文献通读清单和真实实验路线。
如果课题可实验，在至少一个真实 experiment artifact 出现之前，不要输出完整论文或最终 docx。
```

## 适用范围

- 中文理工科毕业论文
- 本科与研究生层级论文
- 偏计算机、数据、算法、系统、仿真、公共数据集类课题

不适用场景：

- 只想快速拼装正文、不关心真实性
- 明知缺乏证据仍要求直接交最终成品



## 隐私与泛化

该公开仓库仅保留通用 skill 内容，不包含学校名称、个人姓名、学号、导师信息或项目私有产物。学校特定 profile、用户样例和本地产物应在各自私有工作区维护，不应硬编码进核心 skill。
